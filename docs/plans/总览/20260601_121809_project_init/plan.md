# 项目初始化与后续开发基线计划

## 目标

建立当前 fork 的项目认知、开发入口、验证命令和风险边界，让后续每个功能都能从稳定上下文开始，而不是每次重新摸仓库。

## 阶段 1：项目基线认知

- 目标：明确仓库定位、主要技术栈、入口文件、验证命令和文档可信度。
- 主要产出：`context.md`、`status.md`。
- 前置依赖：已在 `product/zhanzi` 上建立 fork 工作流规则。
- 验证方式：检查文件证据是否都能在仓库中追溯。

## 阶段 2：后续功能计划机制

- 目标：建立 `feature_list.json` 作为后续功能推进真源。
- 主要产出：`feature_list.json`、`task_list.md`。
- 前置依赖：阶段 1 完成。
- 验证方式：后续新增功能时先追加 `Fxxx`，并为每个功能写明验收和验证命令。

## 阶段 3：专项实施前准入

- 目标：对高风险改动建立准入要求，避免直接在业务代码里试错。
- 主要产出：新增功能前的专项计划或任务条目。
- 前置依赖：明确具体功能需求。
- 验证方式：每个编码型功能都包含“注释补齐与规范自检”、自动化验证命令和未验证风险说明。

## 后续默认流程

1. 从 `product/zhanzi` 切 `codex/<feature-slug>`。
2. 先读本目录 `status.md` 和 `feature_list.json`。
3. 新需求先追加或更新计划条目，再改代码。
4. 修改后运行与改动范围匹配的测试、构建或 lint。
5. 验证通过后合回 `product/zhanzi`。
